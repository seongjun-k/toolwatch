# toolwatch 구현 계획

설계 SSoT는 `계획서.md`(확장 설계는 `확장계획.md`). 이 문서는 코드 구현 단계의 SSoT — 파일 구성, 구현 순서, 위임 대상, 검증 방법만 다룬다. 설계 내용과 충돌 시 계획서가 우선. S1~S6은 기본 시스템(구현 완료), E1~E3은 확장(§6).

## 1. 파일 구성 (최소)

파일 수를 의도적으로 최소화한다. 클래스·패키지 분리는 필요해질 때 한다.

### src/server/ (노트북, Windows)

| 파일 | 책임 |
|---|---|
| `app.py` | Flask 앱 전체: `/frame` 수신·YOLO 추론·재고 대조·디바운스·RFID 세션 귀속·경고 판정·CSV 기록·스냅샷 저장 + 대시보드(`/`)·제어 API. 단일 프로세스, 상태는 메모리 딕셔너리 |
| `config.json` | 설정값: 등록 재고(종별 개수), ROI 좌표, confidence 임계(0.5), 디바운스 프레임 수(3), 미반납 제한(초), RFID 세션 길이(30s), HMI 비밀번호, UID→이름 매핑 |
| `templates/dashboard.html` | 대시보드 겸 제어판 1페이지. CSS는 파일 내 `<style>`, 폰 대응은 미디어쿼리만 |
| `snapshots/` | 반출 확정 프레임 `YYYYmmdd_HHMMSS_공구명.jpg` |
| `logs/events.csv` | `시각,이벤트(OUT/IN/OVERDUE/UNAUTH),공구,UID,이름,스냅샷경로` |

### src/pi/ (라즈베리파이)

| 파일 | 책임 |
|---|---|
| `client.py` | 메인 루프: picamera2 캡처 → 3초 주기 `POST /frame`(JPEG + 최근 태그 UID) → 응답 JSON(`light: green/yellow/red`, `buzzer: off/overdue/unauth`)대로 경광등 구동. RFID 폴링은 스레드 1개, 최근 UID를 변수에 기록 |
| `hw.py` | gpiozero OutputDevice 4채널(적/황/녹/부저) 래핑 + 부저 패턴 2종(미반납: 연속음, 미확인: 단속음). 릴레이 액티브 로우 여부는 상수 1개로 대응 |
| `config.json` | 서버 URL, 캡처 주기, GPIO 핀 번호(릴레이 4채널), RC522 SPI 설정 |

### 그 외

- `model/best.pt` — Colab 학습 산출물 (사용자가 배치)
- 학습·데이터셋은 Colab/Roboflow에서 진행 — 리포에는 `dataset/` 참고 자료만

## 2. HTTP 계약 (Pi ↔ 서버)

- 요청: `POST /frame`, multipart(`image`=JPEG) + form(`uid`=최근 30초 내 태그 UID 또는 빈 문자열)
- 응답: `{"light": "green|yellow|red", "buzzer": "off|overdue|unauth"}`
- Pi는 요청 실패 시 직전 응답 상태 유지, 다음 주기에 재시도 (계획서 3.6)

## 3. 구현 순서 (계획서 4장 일정과 대응)

각 단계는 독립적으로 시연 가능한 상태로 끝낸다. 위임은 CLAUDE.md 기준.

| 단계 | 내용 | 파일 | 위임 | 검증 (호스트) |
|---|---|---|---|---|
| S1 | 서버 골격: `/frame` 수신 → YOLO 추론 → 검출 개수 JSON 응답, `/` 에 최신 프레임 표시 | `app.py`, `dashboard.html`, `config.json` | default-worker | `py_compile` + 로컬에서 curl로 이미지 POST → 응답 확인 (모델은 임시로 사전학습 yolo11n.pt) |
| S2 | 판정 로직: 개수 대조 + 디바운스 + OUT/IN 이벤트 + CSV 기록 + 스냅샷 저장 | `app.py` | default-worker | 판정 함수를 순수 함수로 분리해 `assert` 셀프체크 동봉 (프레임 시퀀스 → 이벤트 시퀀스) |
| S3 | Pi 클라이언트: 캡처 → POST → 경광등 구동, RFID 폴링 스레드 | `client.py`, `hw.py`, `config.json` | default-worker | `py_compile`만 (Pi 전용 모듈은 호스트에 없음). 하드웨어 실행은 사용자 |
| S4 | RFID 귀속 + 경고: 세션(30s, 마지막 태그 우선), UID 귀속, 미반납 타이머, 미확인 반출 판정, 응답에 light/buzzer 반영 | `app.py` | default-worker | 세션·귀속 로직 `assert` 셀프체크 (태그/반출 시퀀스 → 귀속 결과) |
| S5 | 웹 HMI 완성: 상태 표·이력·스냅샷 조회 + 제어(재고·임계시간·경고 해제·UID 매핑), 고정 비밀번호 인증 | `app.py`, `dashboard.html` | default-worker | `py_compile` + 브라우저 수동 확인 (사용자) |
| S6 | 통합 튜닝: 임계값 조정, 시연 시나리오 6장 점검 | 설정값만 | 직접 (설정 변경 수준) | 실기 시연 (사용자) |

- S1~S2가 2주차 마일스톤, S3~S5가 3~4주차. S3은 S1만 있으면 착수 가능 (병렬 가능).
- 각 위임 프롬프트에 포함할 것: 이 문서 §1·§2 해당 부분, 계획서 해당 절 번호, 반환 형식(수정 파일 / diff / 검증 결과 / 남은 우려).

## 4. 핵심 설계 결정 (구현자에게 고정 지시)

- 서버 상태(재고, 디바운스 카운터, 세션, 경고)는 **메모리 보관, DB 없음**. 서버 재시작 시 초기 상태로 복귀해도 무방 (재고는 config의 등록값 기준으로 다음 프레임에서 재판정됨).
- 프레임 저장은 반출 확정 순간의 프레임 재사용 — 별도 촬영 없음 (계획서 3.4).
- 경광등 우선순위 적>황>녹, 부저 패턴은 서버가 결정해 응답으로 내려보내고 Pi는 그대로 구동만 한다 (판정 로직은 전부 서버).
- 스레드는 Pi의 RFID 폴링 1개만. 서버는 Flask 단일 스레드 기본값으로 충분 (요청 주기 3초, 클라이언트 1대).
- 인증은 고정 비밀번호 + Flask 세션 쿠키. 그 이상 금지 (계획서 3.5).

## 5. 남은 결정 (구현 중 사용자 확인 필요)

- GPIO 핀 번호 4개 — 잠정 확정: 적5/황6/녹13/부저19 (`src/pi/config.json`). 배선 후 실측 검증만 남음
- 릴레이 모듈 액티브 로우/하이 — 잠정: 액티브 로우(`hw.py` `RELAY_ACTIVE_LOW=True`). 수령 후 안 맞으면 상수 1개만 반전
- ~~공구 4종 클래스명~~ 확정: `nipper/driver/plier/hex_key` (라벨링·config 키 공용, 표시용 한글은 `config.json` `tool_labels`)
- ROI 좌표 — 카메라 고정 후 1회 측정 (`roi: null`이면 전체 프레임으로 동작하므로 그전에도 개발 가능)

# 확장 구현 계획 (E1~E3)

설계 SSoT: `확장계획.md`(스키마·라우트·발송 규칙 상세). 여기서는 코드 구현 단계만. Pi 코드(`src/pi/`)는 전 단계 무변경.

## 6. 파일 구성 (추가분)

| 파일 | 단계 | 책임 |
|---|---|---|
| `src/server/db.py` | E1 | sqlite3 연결·스키마 초기화(`init_db`)·CRUD 함수. 스키마 DDL은 `확장계획.md` §E1이 SSoT — 여기 복붙하지 않는다 |
| `src/server/toolwatch.db` | E1 | DB 파일 (gitignore 대상, 스키마는 코드로 생성) |
| `src/server/templates/student.html` | E2 | 학생용 모바일 1페이지: 로그인 + 내 대여 목록 + 기한 설정(`datetime-local`). E3에서 알림 켜기 버튼 추가 |
| `src/server/static/sw.js` | E3 | 서비스 워커: push 이벤트 수신 → `showNotification` |
| `src/server/push.py` | E3 | pywebpush 발송 래퍼(`notify(uid, msg)`) + 리마인더 선별 순수 함수(`select_notices`) + 60초 주기 스케줄러 스레드 |

## 7. 구현 순서

| 단계 | 내용 | 파일 | 위임 | 검증 (호스트) |
|---|---|---|---|---|
| E1 | DB 전환: `db.py` 신설, `append_event`→DB, OUT/IN 시 loans INSERT/UPDATE, 기동 시 `returned_at IS NULL`로 rented 복원, `uid_names`→users 이관(대시보드 UID UI 포함) | `db.py`, `app.py` | default-worker | `py_compile` + selfcheck 확장: OUT→IN→OVERDUE 시퀀스가 loans/events 기대 행으로 남는지 assert(임시 파일 DB 사용). 재시작 복원은 selfcheck에서 재연결로 검증 |
| E2 | 학생 웹: `/me`(로그인)·`/me/loans`(목록)·`/me/due`(기한 설정), `session["student_uid"]` 분리, 서버 측 uid 소유 검증, `check_overdue`가 `due_at` 우선 적용 | `student.html`, `app.py` | default-worker | `py_compile` + selfcheck: due_at 설정/미설정 loan의 OVERDUE 판정 분기 assert. 폰 화면 확인은 사용자 |
| E3 | 웹 푸시: VAPID 키 생성 절차 문서화, 구독 등록 라우트(`POST /me/subscribe`), OUT/IN notify 훅(try/except, 실패는 로그만), `select_notices` 순수 함수 + 스케줄러 스레드, `sent_notices` 중복 방지 | `push.py`, `sw.js`, `app.py`, `student.html` | default-worker | `py_compile` + selfcheck: `select_notices`에 시각 주입해 morning/due/overdue 선별·중복 방지 assert. 실 푸시 수신은 사용자(폰) |

- E1→E2→E3 순차 (각각 앞 단계 스키마에 의존). 단계별 독립 커밋.
- 위임 프롬프트에 포함할 것: `확장계획.md` 해당 절 + 본 문서 §6·§8 + 반환 형식(수정 파일 / diff / 검증 결과 / 남은 우려).

## 8. 핵심 설계 결정 (구현자에게 고정 지시)

- **DB 접근은 기존 `_state_lock` 안에서만** (연결은 요청마다 열고 닫기). 트래픽이 3초 주기 1클라이언트라 커넥션 풀·WAL 튜닝 금지.
- 판정 순수 함수(`judge_tools` 등)는 **I/O 없음 원칙 유지** — DB 읽기/쓰기는 전부 호출부(`receive_frame` 등)에서. `check_overdue`·`select_notices`도 데이터를 인자로 받는 순수 함수로.
- CSV 코드는 E1에서 **삭제** (병행 기록 금지). 기존 `logs/events.csv`는 이관하지 않고 보존만.
- 푸시 발송 실패가 판정·응답 흐름을 절대 막지 않는다 (try/except + 로그, 재시도 없음). 410 Gone 응답 구독은 삭제.
- 학생 인증은 UID+이름 일치 확인 수준 (계획서 §3.5의 최소 인증 원칙 연장). 단, loan 소유 검증(`loans.uid == session uid`)은 서버 측에서 생략 불가.
- HTTPS: 개발은 `localhost` 예외로 진행, 폰 시연용 인증서(mkcert 사설 인증서 또는 Flask `ssl_context="adhoc"`)는 E3 착수 시 확정.

## 9. 남은 결정 (E1~E3 착수 전/중 확인)

- VAPID 키쌍 생성·보관 위치 (config.json vs 별도 파일 — secret_key와 함께 시연 전 교체 대상)
- 아침 리마인더 시각 기본값 09:00 유지 여부
- 시연 기기 iOS 여부 (iOS면 홈 화면 추가 사전 안내 필요)
- 기존 CSV 이력의 DB 이관 여부 (기본: 안 함)
