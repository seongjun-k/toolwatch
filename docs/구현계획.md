# toolwatch 구현 계획

설계 SSoT는 `계획서.md`. 이 문서는 코드 구현 단계의 SSoT — 파일 구성, 구현 순서, 위임 대상, 검증 방법만 다룬다. 설계 내용과 충돌 시 계획서가 우선.

## 1. 파일 구성 (최소)

파일 수를 의도적으로 최소화한다. 클래스·패키지 분리는 필요해질 때 한다.

### src/server/ (노트북, Windows)

| 파일 | 책임 |
|---|---|
| `app.py` | Flask 앱 전체: `/frame` 수신·YOLO 추론·재고 대조·디바운스·RFID 세션 귀속·경고 판정·CSV 기록·스냅샷 저장 + 대시보드(`/`)·제어 API. 단일 프로세스, 상태는 메모리 딕셔너리 |
| `config.json` | 설정값: 등록 재고(종별 개수), ROI 좌표, confidence 임계(0.5), 디바운스 프레임 수(3), 미반납 제한(초), RFID 세션 길이(30s), HMI 비밀번호, UID→이름 매핑 |
| `templates/dashboard.html` | 대시보드 겸 제어판 1페이지. CSS는 파일 내 `<style>`, 폰 대응은 미디어쿼리만 |
| `snapshots/` | 반출 확정 프레임 `YYYYmmdd_HHMMSS_공구명.jpg` |
| `logs/events.csv` | `시각,이벤트(OUT/IN/OVERDUE/UNAUTH),공구,UID,이름,스냅샷경로` |

### src/pi/ (라즈베리파이)

| 파일 | 책임 |
|---|---|
| `client.py` | 메인 루프: picamera2 캡처 → 3초 주기 `POST /frame`(JPEG + 최근 태그 UID) → 응답 JSON(`light: green/yellow/red`, `buzzer: off/overdue/unauth`)대로 경광등 구동. RFID 폴링은 스레드 1개, 최근 UID를 변수에 기록 |
| `hw.py` | gpiozero OutputDevice 4채널(적/황/녹/부저) 래핑 + 부저 패턴 2종(미반납: 연속음, 미확인: 단속음). 릴레이 액티브 로우 여부는 상수 1개로 대응 |
| `config.json` | 서버 URL, 캡처 주기, GPIO 핀 번호(릴레이 4채널), RC522 SPI 설정 |

### 그 외

- `model/best.pt` — Colab 학습 산출물 (사용자가 배치)
- 학습·데이터셋은 Colab/Roboflow에서 진행 — 리포에는 `dataset/` 참고 자료만

## 2. HTTP 계약 (Pi ↔ 서버)

- 요청: `POST /frame`, multipart(`image`=JPEG) + form(`uid`=최근 30초 내 태그 UID 또는 빈 문자열)
- 응답: `{"light": "green|yellow|red", "buzzer": "off|overdue|unauth"}`
- Pi는 요청 실패 시 직전 응답 상태 유지, 다음 주기에 재시도 (계획서 3.6)

## 3. 구현 순서 (계획서 4장 일정과 대응)

각 단계는 독립적으로 시연 가능한 상태로 끝낸다. 위임은 CLAUDE.md 기준.

| 단계 | 내용 | 파일 | 위임 | 검증 (호스트) |
|---|---|---|---|---|
| S1 | 서버 골격: `/frame` 수신 → YOLO 추론 → 검출 개수 JSON 응답, `/` 에 최신 프레임 표시 | `app.py`, `dashboard.html`, `config.json` | default-worker | `py_compile` + 로컬에서 curl로 이미지 POST → 응답 확인 (모델은 임시로 사전학습 yolo11n.pt) |
| S2 | 판정 로직: 개수 대조 + 디바운스 + OUT/IN 이벤트 + CSV 기록 + 스냅샷 저장 | `app.py` | default-worker | 판정 함수를 순수 함수로 분리해 `assert` 셀프체크 동봉 (프레임 시퀀스 → 이벤트 시퀀스) |
| S3 | Pi 클라이언트: 캡처 → POST → 경광등 구동, RFID 폴링 스레드 | `client.py`, `hw.py`, `config.json` | default-worker | `py_compile`만 (Pi 전용 모듈은 호스트에 없음). 하드웨어 실행은 사용자 |
| S4 | RFID 귀속 + 경고: 세션(30s, 마지막 태그 우선), UID 귀속, 미반납 타이머, 미확인 반출 판정, 응답에 light/buzzer 반영 | `app.py` | default-worker | 세션·귀속 로직 `assert` 셀프체크 (태그/반출 시퀀스 → 귀속 결과) |
| S5 | 웹 HMI 완성: 상태 표·이력·스냅샷 조회 + 제어(재고·임계시간·경고 해제·UID 매핑), 고정 비밀번호 인증 | `app.py`, `dashboard.html` | default-worker | `py_compile` + 브라우저 수동 확인 (사용자) |
| S6 | 통합 튜닝: 임계값 조정, 시연 시나리오 6장 점검 | 설정값만 | 직접 (설정 변경 수준) | 실기 시연 (사용자) |

- S1~S2가 2주차 마일스톤, S3~S5가 3~4주차. S3은 S1만 있으면 착수 가능 (병렬 가능).
- 각 위임 프롬프트에 포함할 것: 이 문서 §1·§2 해당 부분, 계획서 해당 절 번호, 반환 형식(수정 파일 / diff / 검증 결과 / 남은 우려).

## 4. 핵심 설계 결정 (구현자에게 고정 지시)

- 서버 상태(재고, 디바운스 카운터, 세션, 경고)는 **메모리 보관, DB 없음**. 서버 재시작 시 초기 상태로 복귀해도 무방 (재고는 config의 등록값 기준으로 다음 프레임에서 재판정됨).
- 프레임 저장은 반출 확정 순간의 프레임 재사용 — 별도 촬영 없음 (계획서 3.4).
- 경광등 우선순위 적>황>녹, 부저 패턴은 서버가 결정해 응답으로 내려보내고 Pi는 그대로 구동만 한다 (판정 로직은 전부 서버).
- 스레드는 Pi의 RFID 폴링 1개만. 서버는 Flask 단일 스레드 기본값으로 충분 (요청 주기 3초, 클라이언트 1대).
- 인증은 고정 비밀번호 + Flask 세션 쿠키. 그 이상 금지 (계획서 3.5).

## 5. 남은 결정 (구현 중 사용자 확인 필요)

- GPIO 핀 번호 4개 (릴레이 채널 배선 후 확정 → `src/pi/config.json`)
- 릴레이 모듈 액티브 로우/하이 (모듈 수령 후 확인)
- 공구 4종 클래스명 확정 (라벨링 시점)
- ROI 좌표 (카메라 고정 후 1회 측정)
