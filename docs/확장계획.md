# toolwatch 확장 계획 — DB · 학생용 웹 · 웹 푸시 알림

기존 S1~S5(카메라 판정·RFID 귀속·경광등·대시보드) 위에 얹는 확장 3단계.
SSoT: `docs/계획서.md`(기본 시스템), 본 문서(확장). 구현 대상은 전부 서버(`src/server/`) — Pi 코드는 변경 없음.

제약: 텔레그램 등 외부 메신저 사용 불가(학교 정책). 알림은 자체 서버의 Web Push(VAPID)로만 발송.

---

## E1. SQLite DB (users / loans / events)

### 목표
CSV(`logs/events.csv`) 단일 파일 기록을 SQLite DB로 대체·확장하고, 재시작 시 사라지던 대여 중 상태(`state["rented"]`)를 영속화한다.

### 스키마 (`src/server/toolwatch.db`)

```sql
-- 학생 (기존 config.json의 uid_names를 흡수)
CREATE TABLE users (
  uid          TEXT PRIMARY KEY,          -- RFID UID
  name         TEXT NOT NULL,
  created_at   TEXT NOT NULL DEFAULT (datetime('now','localtime'))
);

-- 대여 (진행 중 + 완료 이력 모두 보관, returned_at IS NULL = 대여 중)
CREATE TABLE loans (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  tool          TEXT NOT NULL,
  uid           TEXT REFERENCES users(uid),  -- NULL = 미확인 반출
  out_at        TEXT NOT NULL,
  due_at        TEXT,                        -- 반납 기한 (E2에서 학생이 설정, NULL = 기본값 overdue_sec 적용)
  returned_at   TEXT,
  overdue_logged INTEGER NOT NULL DEFAULT 0,
  unauth        INTEGER NOT NULL DEFAULT 0,
  cleared       INTEGER NOT NULL DEFAULT 0,
  snapshot_path TEXT
);

-- 이벤트 로그 (기존 CSV 1:1 대응 + loan 연결)
CREATE TABLE events (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  ts        TEXT NOT NULL DEFAULT (datetime('now','localtime')),
  type      TEXT NOT NULL,                 -- OUT / IN / OVERDUE / UNAUTH
  tool      TEXT NOT NULL,
  uid       TEXT,
  loan_id   INTEGER REFERENCES loans(id),
  snapshot_path TEXT
);
```

### 구현 방침
- Python 내장 `sqlite3` 사용, 의존성 추가 없음. `src/server/db.py` 신설(연결·스키마 초기화·CRUD 함수).
- `app.py` 변경 지점:
  - `append_event()` → `db.add_event()` 호출로 교체 (CSV는 폐기, 대시보드 이력은 `events` 테이블 조회).
  - OUT 확정 시 `loans` INSERT, IN 확정 시 해당 공구의 가장 오래된 미반납 loan에 `returned_at` UPDATE (현행 큐 동작 유지).
  - `state["rented"]` 메모리 큐 → 서버 기동 시 `loans WHERE returned_at IS NULL`로 복원 (재시작 내성 확보).
  - `CONFIG["uid_names"]` → `users` 테이블로 이관. 대시보드의 UID 등록/삭제 UI는 DB에 쓰도록 변경.
- Flask 멀티스레드 대비: 요청당 연결 열기(`sqlite3.connect`, `check_same_thread` 이슈 회피) 또는 기존 `_state_lock` 안에서만 접근. 트래픽이 3초 주기 1클라이언트라 후자로 충분.

### 완료 기준 (exit proof)
- `python -m py_compile` 통과 + `--selfcheck` 확장: OUT→IN→OVERDUE 시퀀스가 loans/events에 기대 행으로 남는지 assert.
- 서버 재시작 후 대여 중 목록이 대시보드에 그대로 복원되는지 확인.

## E2. 학생용 모바일 웹페이지 (내 대여 현황 + 반납 기한 설정)

### 목표
학생이 폰 브라우저로 접속해 ① 자기 대여 현황 확인 ② 건별 반납 기한(`due_at`) 설정. 관리자 대시보드(`/`)와 분리된 학생용 화면.

### 라우트 설계
| 라우트 | 기능 |
|---|---|
| `GET /me` | 학생 로그인 화면(등록된 RFID UID + 이름으로 본인 확인 — 별도 비밀번호 없음, 4주 과제 수준의 인증) |
| `GET /me/loans` | 내 대여 목록: 공구명, 대여 시각, 기한, 연체 여부 |
| `POST /me/due` | 특정 loan의 `due_at` 설정/변경 (`<input type="datetime-local">` 사용, 커스텀 위젯 없음) |

### 구현 방침
- `templates/student.html` 1파일 (모바일 우선 단일 페이지, 대시보드 CSS 재사용 가능하면 재사용).
- 세션 분리: 관리자 `session["logged_in"]`과 별개로 `session["student_uid"]` 사용. 학생은 자기 loan만 조회/수정 가능(서버 측에서 uid 일치 검증 — 신뢰 경계이므로 생략 불가).
- 기한 미설정 loan은 기존 `overdue_sec` 기본값으로 연체 판정, 설정 시 `due_at` 우선. → `check_overdue()`에 `due_at` 인자 반영 (순수 함수 시그니처 확장, selfcheck 케이스 추가).

### 완료 기준
- selfcheck: `due_at` 설정 loan은 `due_at` 기준, 미설정 loan은 `overdue_sec` 기준으로 OVERDUE 판정되는지 assert.
- 폰 브라우저에서 로그인 → 목록 확인 → 기한 변경이 DB에 반영되는지 확인 (사용자 실기).

## E3. 웹 푸시 알림 (즉시 알림 + 스케줄러)

### 목표
학생 폰으로 푸시 알림 4종:
1. **대여 즉시** — 내 UID로 OUT 확정 시 "니퍼 대여 처리됨"
2. **반납 즉시** — IN 확정 시 "니퍼 반납 완료"
3. **아침 리마인더** — 반납 기한이 오늘인 loan 보유자에게 아침(기본 09:00) 1회
4. **기한 도래/연체 경고** — `due_at` 도달 시 1회, 이후 미반납 지속 시 경고 1회

### 기술 구성
- **Web Push (VAPID)**: 외부 서비스 계정 불필요, 자체 키쌍으로 브라우저 푸시 서버(FCM/Mozilla 엔드포인트)에 직접 발송. 의존성 `pywebpush` 1개 추가 (`pip install pywebpush`).
- 프런트: `static/sw.js`(서비스 워커) + `/me` 페이지에 "알림 켜기" 버튼 → `PushManager.subscribe()` → 구독 정보를 서버로 POST.
- DB 추가 테이블:

```sql
CREATE TABLE push_subscriptions (
  uid       TEXT NOT NULL REFERENCES users(uid),
  endpoint  TEXT NOT NULL,
  keys_json TEXT NOT NULL,     -- p256dh/auth
  PRIMARY KEY (uid, endpoint)  -- 학생 1명이 여러 기기 등록 가능
);

-- 리마인더 중복 발송 방지 (아침/기한/연체 각 1회)
CREATE TABLE sent_notices (
  loan_id INTEGER NOT NULL REFERENCES loans(id),
  kind    TEXT NOT NULL,       -- morning / due / overdue
  PRIMARY KEY (loan_id, kind)
);
```

### 발송 지점
- 즉시 알림(1·2): `app.py`의 OUT/IN 확정 처리 직후 `notify(uid, message)` 호출. 발송 실패는 로그만 남기고 본 판정 흐름에 영향 없음 (try/except — 푸시는 부가 기능).
- 스케줄러(3·4): 별도 데몬 스레드 1개, 60초 주기 루프로 DB 조회 후 발송 (`threading.Timer`/`while+sleep`, APScheduler 등 추가 의존성 불필요).
  - morning: `date(due_at) = today AND 현재시각 >= morning_time` → 미발송이면 발송.
  - due: `now >= due_at` → 1회. overdue: due 발송 후에도 미반납이면 경고 1회.
- config 추가 키: `vapid_public_key`, `vapid_private_key`, `vapid_email`, `morning_notify_time`("09:00").

### 제약·전제
- **HTTPS 필수**: 서비스 워커/푸시는 secure context에서만 동작. 로컬 개발은 `localhost` 예외로 가능하나, 폰에서 쓰려면 자체 서명 인증서(`ssl_context`) 또는 `mkcert`로 사설 인증서 발급 필요. 시연 환경(학교 내부망)에서의 인증서 처리 방법은 구현 시점에 확정.
- iOS Safari는 홈 화면 추가(PWA 설치) 후에만 웹 푸시 허용 — 시연 기기가 iPhone이면 사전 설치 안내 필요.

### 완료 기준
- 발송 판정 로직(morning/due/overdue 선별 쿼리 + 중복 방지)을 순수 함수로 분리해 selfcheck에 시각 주입 케이스 추가.
- 실기: 폰에서 구독 → RFID 대여 → 푸시 수신 확인 (사용자 실기).

## 진행 순서·규모

| 단계 | 신규/수정 파일 | 예상 규모 | 선행 조건 |
|---|---|---|---|
| E1 DB | `db.py`(신규), `app.py`(수정) | ~200줄 | 없음 |
| E2 학생 웹 | `student.html`(신규), `app.py`(라우트 3개), `check_overdue` 확장 | ~150줄 | E1 |
| E3 웹 푸시 | `sw.js`+`push.py`(신규), `app.py`(발송 훅+스레드), `student.html`(구독 버튼) | ~200줄 | E1, E2, HTTPS |

- 각 단계는 독립 커밋. E1·E2는 호스트에서 완결 검증 가능, E3만 실기(폰) 검증 필요.
- 규모상 각 단계 구현은 default-worker 위임 대상(4파일/50줄+ 기준), 메인이 리뷰.
