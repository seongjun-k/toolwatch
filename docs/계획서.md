# 실습실 공구 반납 감시대 — 프로젝트 계획서

임베디드시스템 4주 프로젝트 · 작성일 2026-07-15

## 1. 배경 및 목표

### 1.1 문제 정의
실습실 공용 공구는 사용 후 제자리에 반납되지 않거나 분실되는 일이 잦다. 누가 언제 가져갔는지 기록이 없어 관리자가 수기로 확인해야 하고, 분실 시 책임 소재를 알 수 없다. 상용 스마트 공구 캐비닛(henchmanTRAK kabTRAK 등)이 존재하지만 고가라 실습실 도입이 어렵다.

### 1.2 목표
카메라 1대 + RFID 리더 + 라즈베리파이로 구성된 **저비용 공구 반납 감시 시스템** 구축:

1. YOLO 객체 검출로 거치대의 공구 재고를 자동 파악 (무엇이 몇 개)
2. RFID 태그로 대여자 식별 (누가)
3. 반출/반납 시각 자동 기록 (언제)
4. 제한 시간 초과 미반납 → 부저/LED + 대시보드 경고
5. RFID 태그 없이 공구 반출 → 즉시 "미확인 반출" 경고 + 반출 순간 스냅샷 저장 (증거 확보)
6. 웹 HMI로 같은 네트워크 내 PC·휴대폰 어디서든 상태 조회 및 제어

### 1.3 차별점
비전 재고 감시(Edge Impulse Workplace Organizer)와 RFID 대여 귀속(Etsy rfid-checkout)이 각각 선행 사례로 존재하나, 둘을 결합해 도난 감시까지 수행하는 저비용 통합 시스템은 없음. 선행 사례 상세: `레퍼런스.md`

## 2. 시스템 구성

### 2.1 아키텍처

```
┌─ 라즈베리파이 (엣지) ─────────┐         ┌─ 노트북 (서버) ──────────────┐
│ Pi 카메라  ──캡처──┐          │  HTTP   │ /frame 수신                  │
│ RC522 RFID (SPI) ──┼─ 클라이언트 ─POST─→ │  → YOLO11n 추론              │
│ 3색 경광등 (릴레이 경유) ←┘    │ ←응답─  │  → 재고 대조·디바운스         │
└──────────────────────────────┘  JSON   │  → RFID 세션 귀속            │
                                          │  → CSV 로그                  │
                                          │ Flask 대시보드 (/)           │
                                          └──────────────────────────────┘
```

- **라즈베리파이**: 3초 주기로 프레임 캡처 → 노트북에 HTTP POST (RFID 태그 UID 동봉) → 응답 JSON의 경고 명령으로 부저/LED 구동
- **노트북**: YOLO 추론, 재고 판정, 대여자 귀속, 로그, 대시보드
- **통신**: HTTP 단일 채널 (요청에 프레임+UID, 응답에 경고 명령) — Pi에 서버 불필요, 노트북 IP만 알면 동작. ROS/MQTT 등 미들웨어는 주기적 요청-응답 패턴에 불필요하여 배제

### 2.2 하드웨어 (BOM)

| 품목 | 용도 | 비고 |
|---|---|---|
| Raspberry Pi (4 이상) | 엣지 제어 | 보유 장비 |
| Pi 카메라 모듈 | 거치대 촬영 | 거치대 정면 40~50cm 고정 |
| RFID RC522 + 카드/태그 | 대여자 식별 | SPI 연결, 약 2천원 |
| 3색 경광등 (적/황/녹 + 내장 부저, 공통 -, 입력 12V) | 상태 표시·경고음 | GPIO 직결 불가(12V) → 릴레이 모듈 경유 |
| 릴레이 모듈 (4채널: 적/황/녹/부저) | 경광등 12V 라인 스위칭 | GPIO(3.3V 신호)로 릴레이 제어, 12V는 별도 외부 전원 공급 |
| 12V 외부 전원 (아답터) | 경광등 구동 + Pi 전원 원천 | GND는 공통 |
| 스텝다운 컨버터 (12V→5V) | 라즈베리파이 전원 공급 | 12V 단일 어댑터에서 분기, 5V/3A 이상 출력 확보 |
| 네트망 300×300mm | 공구 거치대 | 공구 4종, 간격 30mm 이상 |
| 폼보드 (단색 배경판) | 배경 노이즈 제거 | 네트망 뒤 부착 |
| 공구 4종 | 감시 대상 | 니퍼, 드라이버, 펜치, 육각렌치 (클래스명: nipper/driver/plier/hex_key) |

### 2.4 기구 설계 (제작물)

거치대 하나가 아니라 아래 요소 전체를 하나의 구조물로 통합 제작한다:

- **공구 거치대**: 네트망 300×300mm + 폼보드 배경판, 공구 4종 간격 30mm 이상 배치
- **카메라 고정대**: 거치대 정면 40~50cm, 화각이 네트망 전체를 담도록 높이·각도 고정 (흔들림 방지가 검출 안정성의 전제)
- **경광등·릴레이 마운트**: 경광등은 시야에 잘 띄는 상단, 릴레이 모듈은 Pi 인근 고정
- **RFID 리더 거치**: 사용자가 자연스럽게 태그하는 높이(거치대 측면)에 고정
- **전원·배선 정리**: 12V 어댑터 1개 → 경광등(릴레이 경유) + 스텝다운 컨버터(→Pi 5V) 분기, GND 공통. SPI·GPIO 배선은 케이블 타이로 고정해 시연 중 접촉 불량 방지

### 2.3 소프트웨어 스택

| 구분 | 선택 | 근거 |
|---|---|---|
| 검출 모델 | **YOLO11n** (ultralytics) | 자료 최다, nano로 4클래스·고정카메라·단색배경이면 충분. 부족 시 11s로 승급 |
| 학습 | 로컬 GPU (CUDA) | 11n·4클래스·100~150장 기준 10~20분, 세션 끊김 없음 |
| 라벨링 | Roboflow 무료 플랜 | 공개 공구 데이터셋 병용 가능 |
| 추론·서버 | 노트북 Python + Flask | Pi 최적화(NCNN 등) 불필요, 개발·디버깅 속도 확보 |
| Pi 측 | Python (picamera2, requests, gpiozero, mfrc522) | 경광등은 gpiozero OutputDevice로 릴레이 채널(적/황/녹/부저) ON/OFF 제어 |
| 로그 | CSV → SQLite | 초기 구현은 CSV, 확장 단계(E1)에서 SQLite로 전환 (`확장계획.md`) |

## 3. 상세 설계

### 3.1 YOLO 학습
- 클래스 4개 (공구 4종). 비슷한 공구(십자/일자 드라이버 등)는 한 클래스로 통합해 난이도 절감
- 데이터: **실제 설치 위치·조명·배경판에서** 종별 최소 60장(하한), 목표 100~150장 촬영 (걸린 상태/빠진 상태/손 개입/조명 변화 포함) + Roboflow 공개 공구 데이터셋 보강
- 검증: hold-out 20%로 mAP 확인, 실환경 오검출은 3주차에 데이터 추가·재학습으로 보정

### 3.2 판정 알고리즘
1. **ROI 크롭**: 공구판 영역만 잘라 추론 — 배경(사람, 책상) 오검출 원천 차단. 설치 시 좌표 1회 지정
2. **개수 기반 판정**: `등록 재고(종별 개수) − 현재 검출 개수 = 대여 중`. 위치 매칭 불필요 → 아무 자리에 걸어도 동작
3. **디바운스**: 3프레임 연속(약 9초) 동일 판정일 때만 반출/반납 확정 — 단발 오검출·손 가림에 강건. 오탐 잦으면 5프레임으로 조정
4. **미반납 경고**: 반출 확정 후 제한 시간(기본 2시간, 설정값) 초과 시 경고

경광등 색상 매핑 (3색 활용 — 황색으로 "대여 중(정상)" 상태를 별도 표시):

| 색상 | 상태 | 부저 |
|---|---|---|
| 녹색 | 전 공구 재고 정상 | 없음 |
| 황색 | 정상 대여 중 (제한 시간 이내) | 없음 |
| 적색 | 미반납 초과 또는 미확인 반출 | 울림 (미확인 반출은 경고음 패턴을 다르게 하여 구분) |

여러 공구가 서로 다른 상태일 때는 **우선순위 적 > 황 > 녹** — 가장 심각한 상태 하나로 표시 (예: A 정상 대여 중 + B 미반납 초과 → 적색)

### 3.3 RFID 대여자 귀속
```
카드 태그 → UID 전송 → 30초 "세션" 열림
세션 중 반출/반납 확정된 공구 → 해당 UID에 귀속
세션 외 반출 확정 → "미확인 반출": 즉시 경광등 적색+부저(미확인 반출 패턴) + 대시보드 경고 + 반출 확정 프레임 스냅샷 저장
```
- UID ↔ 이름(학번) 매핑: 사전 등록 (시연용 팀원 카드 등록, JSON 파일)
- 세션 30초는 Etsy rfid-checkout의 60초 패턴 참고, 시연 중 조정 가능하게 설정값으로
- 세션 중 다른 카드 태그 시 **기존 세션 즉시 종료 + 새 UID로 세션 갱신** (마지막 태그 우선) — 연속 태그 시 앞사람에게 잘못 귀속되는 것 방지

### 3.4 미확인 반출 증거 저장
- 디바운스로 반출이 확정되는 순간의 프레임을 `server/snapshots/`에 `시각_공구명.jpg`로 저장 (RFID 세션 유무와 무관하게 모든 반출에 저장, 미확인 반출은 대시보드에서 별도 강조 표시)
- CSV 로그에 스냅샷 파일 경로 컬럼 추가 → 대시보드 이력에서 클릭하면 해당 사진 확인
- 저장은 이미 확정된 프레임 재사용이라 추가 촬영·연산 없음 (판정 로직에 얹는 부가 동작)

### 3.5 웹 HMI (대시보드 겸 제어판)
- Flask 페이지를 조회 전용에서 **제어 포함**으로 확장, 같은 네트워크(실습실 Wi-Fi/공유기) 내 어떤 기기에서든 `http://<노트북IP>:5000` 접속
- 조회: 공구별 상태(재고/대여자/반출 시각/경과 시간), 최신 검출 이미지, 미확인 반출 스냅샷 이력
- 제어: 등록 재고 수정, 미반납 임계 시간 설정, 경고 수동 해제, RFID UID-이름 매핑 등록/삭제
- 인증: 실습실 내부망 한정이라 최소 수준(고정 비밀번호 1개, Flask 세션 쿠키)으로 충분 — 외부 노출 시나리오 아니므로 OAuth 등 불필요
- 반응형 레이아웃(CSS만, 별도 프레임워크 불필요)으로 폰 화면에서도 표·버튼 사용 가능하게

### 3.6 예외 처리
| 상황 | 동작 |
|---|---|
| 네트워크 단절 | Pi가 요청 실패 시 기존 경고 상태 유지, 3초 후 재시도 |
| 손·몸으로 가림 | 디바운스가 자동 흡수 (연속 판정 조건 미충족) |
| 검출 신뢰도 저하 | confidence 0.5 미만 무시, 임계값 설정값화 |

## 4. 4주 일정

| 주 | 목표 | 세부 작업 | 산출물 (마일스톤) |
|---|---|---|---|
| 1주 | 환경 구축 + 모델 1차 | 거치대 제작(네트망+배경판+공구 배치), 카메라 고정·프레이밍, 데이터 촬영·라벨링, 로컬 GPU 학습, 노트북 추론 확인 | **학습된 모델 + 정지 이미지 검출 데모** |
| 2주 | 재고 판정 파이프라인 | Pi 프레임 전송 클라이언트, Flask `/frame` 수신·추론, 개수 대조+디바운스 판정, CSV 로그 | **공구 빼면 자동으로 OUT 기록되는 데모** |
| 3주 | RFID + 경고 + 웹 HMI(조회) | RC522 연동(UID 읽기·세션), 대여자 귀속, 부저/LED 경고(미반납·미확인 반출), 반출 스냅샷 저장, Flask 웹 HMI **조회 기능**, 오검출 데이터 추가·재학습 | **전체 기능 통합 동작 + 폰에서 접속 확인** |
| 4주 | 통합 검증 + 발표 | 웹 HMI **제어 기능**(재고·임계시간·경고 해제·UID 매핑), 시나리오 테스트(정상 대여/미반납/미확인 반출), 임계값 튜닝(디바운스·confidence·제한시간·세션), 발표자료·시연 영상 | **최종 시연 + 발표** |

주차별 판단 기준: 1주차 종료 시 검출 정확도가 불충분하면 2주차 전반을 데이터 보강에 재배정 (아래 리스크 대응).

## 5. 리스크 및 대응

| 리스크 | 영향 | 대응 |
|---|---|---|
| YOLO 검출 정확도 부족 | 전체 기능 불성립 | **1주차에 학습 완료해 조기 판정** → 2주차에 데이터 추가 여유 확보. 유사 공구 클래스 통합 |
| 조명·배경 변화 | 오검출 증가 | 실환경 데이터 촬영, 단색 배경판, ROI 크롭, confidence 튜닝 |
| 최악의 경우 | 종류 인식 실패 | 클래스 구분 포기 → "공구 있음" 단일 클래스 + 자리(슬롯) 기반 판별로 후퇴 |
| RFID 연동 지연 | 귀속 기능 누락 | 세션 로직이 독립 모듈이라 재고 감시만으로도 시연 가능. 최후 보루: USB 키패드 학번 입력 대체 |
| 네트워크 단절 | 감시 공백 | 재시도 + 경고 상태 유지, 대시보드에 "연결 끊김" 표시 |
| RFID 세션 도용 | 무단 반출이 정상 대여로 오귀속 | **본 범위에서는 감수.** A가 태그해 세션이 열린 30초 안에 B가 다른 공구를 집으면 개수 기반 판정으로는 구분 불가 — 위치·인물 정보를 안 쓰는 설계의 원리적 한계(§3.3). 완화는 `rfid_session_sec` 30 → 10~15초 단축뿐이며, 근본 해결(공구별 슬롯 매칭 또는 인물 추적)은 확장 과제 |

## 6. 평가(시연) 시나리오

1. **정상 대여·반납**: 카드 태그 → 니퍼 반출 → 대시보드에 "니퍼 OUT / 홍길동 / 14:32" → 반납 → 자동 해제·기록
2. **미반납 경고**: 제한 시간(시연용 1분으로 단축) 초과 → 부저 + 적색 LED + 대시보드 경고
3. **미확인 반출 (도난 감시)**: 태그 없이 공구 반출 → 즉시 경고 + 반출 순간 스냅샷이 대시보드 이력에 표시
4. **강건성**: 손으로 공구판 가리기, 공구를 다른 자리에 걸기 → 오경고 없음 확인
5. **웹 HMI**: 발표자 폰으로 실습실 Wi-Fi 접속해 대시보드에서 임계 시간 조정·경고 수동 해제 시연

## 7. 확장 구현 (E1~E3) — 상세: `확장계획.md`

기본 시스템(S1~S5) 완성 후 얹는 확장 3단계. 전부 서버 측 구현이며 Pi 코드 변경 없음. 텔레그램 등 외부 메신저는 학교 정책상 사용 불가 → 알림은 자체 서버의 Web Push(VAPID)로 발송.

| 단계 | 내용 | 핵심 |
|---|---|---|
| E1 | SQLite DB | `users`/`loans`/`events` 3테이블. CSV 로그 대체 + `uid_names` 이관 + 재시작 시 대여 중 상태 복원. 내장 `sqlite3`, 의존성 추가 없음 |
| E2 | 학생용 모바일 웹 | `/me` 라우트: 학생이 폰으로 자기 대여 현황 조회 + 건별 반납 기한(`due_at`) 설정. 기한 설정 시 `overdue_sec` 대신 `due_at` 기준으로 연체 판정 |
| E3 | 웹 푸시 알림 | 대여/반납 즉시 알림 + 아침 리마인더·기한 도래·연체 경고(60초 주기 스케줄러, 건별 1회). `pywebpush` 1개 추가, HTTPS 필수(사설 인증서), iOS는 홈 화면 추가 후 푸시 가능 |
